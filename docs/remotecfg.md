# The `remotecfg` path (primary)

## What Alloy needs

A minimal bootstrap `/etc/alloy/config.alloy` with a single `remotecfg` block.
See [`examples/bootstrap.alloy`](../examples/bootstrap.alloy).

```alloy
remotecfg {
  url          = "https://fleet.example.com"
  bearer_token = sys.env("FLEET_AGENT_BEARER_TOKEN")
  id           = constants.hostname
  attributes   = { "env" = "prod", "role" = "edge" }
  poll_frequency = "30s"
}
```

Docs for every `remotecfg` field:
<https://grafana.com/docs/alloy/next/reference/config-blocks/remotecfg/>

## Endpoints served

All three are unary Connect RPCs at:

```
POST /collector.v1.CollectorService/GetConfig
POST /collector.v1.CollectorService/RegisterCollector
POST /collector.v1.CollectorService/UnregisterCollector
```

Content negotiation:

- `Content-Type: application/proto` â€” protobuf binary (Alloy's default)
- `Content-Type: application/json` â€” Connect JSON (handy for curl / debugging)

Auth: `Authorization: Bearer <AGENT_BEARER_TOKEN>` on every request.

The `.proto` is vendored verbatim at
[`proto/collector/v1/collector.proto`](../proto/collector/v1/collector.proto).
It is loaded at runtime by `protobufjs` â€” no codegen step.

## Wire examples

### GetConfig (JSON)

Request:

```bash
curl -X POST https://fleet.example.com/collector.v1.CollectorService/GetConfig \
  -H "Authorization: Bearer $AGENT_BEARER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
        "id": "host-01",
        "local_attributes": { "env": "prod", "role": "edge" },
        "hash": ""
      }'
```

Response (first time, hash empty):

```json
{
  "content": "// Generated by Alloy Fleet Manager. ...\nlogging { level = \"info\" }\n...",
  "hash":    "37f2965241c6a86ca5479b917166af2d9b96ac1cfe5d3a75db03ccc952190ef0",
  "notModified": false
}
```

Subsequent request with the same hash:

```json
{ "content": "", "hash": "37f29...", "notModified": true }
```

### RegisterCollector

```bash
curl -X POST https://fleet.example.com/collector.v1.CollectorService/RegisterCollector \
  -H "Authorization: Bearer $AGENT_BEARER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{ "id": "host-01", "name": "host-01.prod", "local_attributes": {"env":"prod"} }'
```

Returns `{}` on success.

## Selector semantics

A pipeline is delivered to a collector **iff** every key/value pair in the
pipeline's `selector` JSON is present (and equal) in the collector's
`local_attributes`.

Examples (pipeline selector on the left, collector attrs on the right):

| Selector                    | Collector attrs                                 | Matches? |
|-----------------------------|-------------------------------------------------|----------|
| `{}`                        | anything                                        | yes      |
| `{"role":"edge"}`           | `{"env":"prod","role":"edge"}`                  | yes      |
| `{"role":"edge","env":"prod"}` | `{"env":"prod","role":"edge","zone":"us"}`    | yes      |
| `{"role":"edge"}`           | `{"role":"gateway"}`                            | no       |

An unmatched collector gets a stub config saying no pipelines matched; Alloy
will load it and log the fact but otherwise do nothing.

## Health / inventory

Every `GetConfig` call updates `remotecfg_collectors.last_seen` and the
last-reported `remote_config_status`. Admin endpoint:

```bash
curl -H "Authorization: Bearer $ADMIN_TOKEN" \
  https://fleet.example.com/remotecfg/collectors
```

## Protocol limitations of this MVP

- Bearer token is **shared** across all collectors. Per-collector tokens are
  future work; the protobuf `RegisterCollector` can be extended to mint them.
- Only Connect **unary** is supported. CollectorService has no streaming
  RPCs, so `application/connect+proto` envelopes are not implemented.
- `GetConfigRequest.effective_config` is accepted on the wire but ignored â€”
  we don't (yet) store the full effective config per collector.
